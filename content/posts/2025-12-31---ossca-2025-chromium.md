지난 5월 말에 6주간 진행한 오픈소스 컨트리뷰션 아카데미 체험형 프로그램 git 활용 및 Chromium 프로젝트의 멘토링을 끝냈다. 마친지 네 달이 다 되어서야 멘토링 과정과 후기를 남긴다.  

## 시작

4월 중순부터 5월말까지 6주 동안 오픈소스 컨트리뷰션 아카데미 체험형 멘토를 했다. 지난 2월에 2023년, 2024년에 오픈 프런티어 기여형을 하면서 연을 맺게 된 오픈업 담당자 분이 내 블로그에서 멘토링도 한 번 해보고 싶다는 내용을 보셨다면서... 멘토를 제안해 주셨다. 체험형은 총 6주동안 진행하는데, 그 중 2주는 git 과 GitHub, 나머지 4주는 프로젝트 내용으로 멘토링 프로그램을 진행한다고 들었다. Chromium 과 같은 대형 프로젝트로 그 짧은 기간에 멘토링을 하는게 가능한지 의문이 들었지만 크게 고민하지 않고 하기로 결정했다. 어쨌든 13주 이상 진행하고 최종 보고회까지 있는 참여형 프로그램보다는 부담이 덜할 것 같았다.

멘토링을 하게 된 첫번째 이유는 멘토링을 하면 나도 Chromium 을 더 관심있게 보고 다시 기여를 열심히 하지 않을까라는 막연한 기대 때문이었다. 그 다음은 현재 하고 있는 일에 대한 아쉬움을 해소하고 싶은 이유가 컸다. 작년에 옮긴 부서에서는 적은 인원이 소프트웨어보다는 조금 더 하드웨어나 다른 종속성이 있는 PoC 개발 위주로 일을 하다 보니 일하는 방식이나 업무의 깊이에 대한 아쉬움이 많았다. 보통은 연차가 쌓일수록 다른 사람들을 리딩하거나 협업할 일이 많이 생기는데 지금까지 그런 기회도 많이 없었어서 관련된 역량도 키우고 싶었다. 회사에서는 요즘 젊은 주니어 개발자들을 보기가 어려운데, 내가 주니어 시절에 오픈소스를 늦게 시작한 것을 아쉽게 생각했기 때문에 주변의 주니어 개발자들이 조금이라도 더 빨리 시작할 수 있게 도움을 주고 싶었다. AI 시대에 취업도 어렵고, 예전과는 다른 목표와 방식으로 일을 하는 것이 중요하게 바뀌고 있는 시점에 이 멘토링 프로그램이 얼마나 효용성이 있는지 의문이 들긴 했지만 조금이라도 도움이 되길 바라며 준비를 시작했다.

## 멘티 선발

4월 초부터 멘티들의 지원을 받았다. 내가 제공된 템플릿에 맞춰 프로젝트 소개 페이지를 직접 만들었고 멘티들은 각 프로젝트 소개 페이지를 참고해서 지원을 했다. 2025년 1차 체험형 프로그램의 프로젝트는 10개였고, 멘티들은 자기소개, 지원동기, 개발경험 등을 작성해서 2지망까지 지원할 수 있었다.

각 프로젝트의 멘토가 지원서 내용을 토대로 멘티를 직접 선발했다. 25~30명 선발로 가이드를 받았는데 그보다는 적은 인원이 1지망으로 Chromium 프로젝트를 지원했다. 멘토 입장에서는 20명 이상은 좀 많다고 생각이 들었지만 그래도 최대한 기회를 주고 싶어서 많은 인원을 뽑으려고 했다. 지원서 내용이 조금 성의없다고 느낀 분들을 빼고 2지망 분들을 추가해서 뽑았다. 1지망 지원서를 대충 쓴 사람보다는 2지망이라도 열심히 쓴 사람을 뽑고 싶었다. 당연히 2지망에서 뽑은 분들은 1지망 지원서도 열심히 썼을 것이기 때문에 (2지망으로 쓴 분들이 1지망을 어디에 어떤 내용으로 썼는지는 알 수 없었다.) 대부분 1지망 프로그램으로 뽑혀가고 1명만 우리 쪽으로 배정이 되었다.

최종 선발 된 멘티는 22명이었다. 발대식 전에 1명이 불참 의사를 밝혀서 발대식 기준 21명으로 시작하였고 그 중 1명을 제외한 20명이 1회 이상 모임에 참여했다.

### 준비와 발대식

지금껏 오픈소스 컨트리뷰션 아카데미나 다른 프로그램으로 멘토링을 해보거나 받아본 경험이 없었다. 어느 정도 커리큘럼과 기본적인 구성안을 만들었지만 실제로 어떻게 진행해야 할지 막막했다. 그래서 미리 모든 걸 준비하긴 어려웠고, 진행하면서 멘티들의 상황이나 목표에 맞추기로 했다. 최대한 많은 인원을 끝까지 끌고 가고 싶었다. 멘티 선발 후 발대식 전에 사전 설문조사를 했다. 이 부분은 정말 잘한 일이라고 생각이 들었다. 지원서를 통해 내가 정보를 어느 정도 알 수 있었지만 프로젝트와 관련된 내용을 좀 더 추가하여 멘티들을 조금 더 깊이 이해하는데 실제로 도움이 되었다. 그 내용을 여러번 확인하면서 멘티들의 얼굴과 이름도 금방 익힐 수 있었다. 이전에 오픈소스 컨트리뷰션 아카데미에 참여한 분들이 몇 명 있어서 좋았던 점과 아쉬웠던 점도 먼저 알아봤다. 경험자는 3~4명 정도 있었는데 공통으로 아쉽게 느꼈던 점이 너무 어려워서 포기하는 분들이 많았고 끝가지 남아있던 분들이 소수였다는 점이었다. 나는 이 부분을 고려해서 최대한 많은 인원을 끝까지 끌고 가고 싶었다. 

프로젝트별로 나눠서 평일 저녁에 발대식을 진행했다. 21명 중 15명이 참석했다. 발대식에서는 먼저 주최 측에서 전체 프로그램 소개와 공지를 하고 이후에 각 강의실에서 어색하게 피자를 먹으며 오리엔테이션을 했다. 사전 설문조사 내용과 앞으로 진행할 내용을 공유하고 모임 시간을 정했다. 멘토 포함 16명 중 MBTI E 성향이 한 명 밖에 없었던 어색한 자리였다. 뒤돌아 봤을때 발대식에 참여한 분들은 대부분 수료를 했다. 가볍게 인사하고 목표를 공유하는 자리가 없다보니 조금 어렵지 않았나 생각이 든다. 

## 모임 진행 방식

6주 동안 매주 오프라인 1회, 온라인 1회 모임을 진행했다. 오프라인 모임을 기본으로 하고 멀리 살아 매주 올라오기 힘든 분들과 불참자 대상으로 온라인 보강을 했다. 오프라인 모임은 매주 일요일과 토요일을 번갈아가면서 일-토-일-토-일-토 모임을 가졌다. 그 다음주 목요일 저녁에 온라인 모임 진행을 했다. 매주 오프라인 모임에는 10~12명, 온라인 모임에는 5~7명 정도가 참여했다.  

### 모임 내용

모임은 강의식으로 진행했다. 강의만 하면 이해하기 어렵고 지루하기 때문에 중간에 실습도 조금 진행했다. 멘티들은 대부분 코로나 시대에 학교를 다녔기 때문에 비대면 강의에 익숙한 세대인데 나 같은 경우는 그게 아니어서 강의하는 입장에서 조금 어색했다. 영상은 나만 켰었고 멘티들은 음성보다는 채팅으로 질문을 했다.

#### 1, 2주차
1, 2주차에는 git 과 GitHub 와 같은 협업 도구에 대한 멘토링이 필수였는데 2주 내내 그 내용만 다루기 보다는 이 때 오픈소스와 Chromium 에 대한 전반적인 내용을 먼저 같이 공유하면

오픈소스 프로젝트 기여 활동에 대한 전반적인 내용을 공유했다. 전반적인 참여 방법과 프로세스, 프로젝트별 특성을 공유하고 그동안 내가 경험했던 것들을 공유했다. 그 다음에는 git 에 대한 강의를 했는데 너무 기본적인 내용보다는 설문조사를 통해 어려워하는 부분과 내가 현업에서 실제로 많이 쓰고 있는 명령어나 코드 전략을 공유했다. 멘티들이 이미 일을 하면서 실무 경험이 있는 경우도 있고 부트캠프를 통해 프로젝트를 많이 해본 것 같아 이렇게 해봤다. 기본적인 내용이 익숙하지 않은 분들도 있긴 했다. 따로 문의를 하셨는데 주최 측에서 git 강의를 따로 유튜브에 업로드하여 멘티들에게 제공을 했기 때문에 이 부분은 참고하게 했다. 

- 오픈소스 활동의 의미, 기여 동기와 얻을 수 있는 가치 공유
    
- 오픈소스 참여 방법과 프로젝트 탐색법 학습
    
- Git 기본 명령어, 브랜치 전략, alias 설정 등 실습
    
- GitHub를 통한 협업 방식, PR 전략, Trunk-based Development 소개
    
- 개인 dotfiles 및 alias 활용법 공유

- 오픈소스 개발 프로세스 전체 흐름 소개 (이슈 → 리뷰 → 머지 → 배포)
    
- Chromium 기여 흐름 및 Google Issue Tracker, Gerrit 리뷰 시스템 소개
    
- Blink, Web Platform, WebIDL 등 주요 기술 요소 및 구조 개요
    
- 웹 표준 문서화 과정 및 Web Platform Test 소개
    
- GitHub 프로젝트 관리, CI/CD, 문서 구성 방식 학습

#### 3주차
3주차부터는 본격적인 멘토링이 들어가야 했다. 어떤 것을 해야할지 고민을 많이 했는데 갑자기 실습할 거리가 생각났다. 

바이브 코딩으로 GitHub Page 기반의 웹페이지를 급하게 만들었다. 5월 초 연휴 기간에 고성 맹그로브에 가서 폐관 수련을 하면서 컨트리뷰션 프로세스를 만들었다. 미리 계획한 것이 아니라 갑자기 떠올라서 진행을 했다. 발표 자료와 강의, 데모로 아무리 보여줘도 직접 해보지 않으면 이해가 되지 않는 부분이 많기 때문에 직접 해보는 것이 중요하다고 생각했다. 특히 Chromium 은 gerrit 을 사용하기 때문에 생소하다. 대부분 생소한 프로세스를 통해 진행되기 때문에 이해하기 어렵다. 또 Chromium Issue Tracker 쪽도 같이 하기 보다는 이슈를 올리거나 이슈를 별도로 요청해서 할당 받지 않고 바로 진행할 수 있는 거리를 찾아서 진행했다. docs 마크다운 문서의 깨진 링크를 수정해서 고치는 실습을 진행했다. 최상위 문서에서는 하나하나 눌러봤고 그 후에 ChatGPT 의 도움을 받아 스크립트를 돌려 깨진 링크를 찾았다.

멘티 인원 수에 맞춰 이슈를 모두 자체 repository 에 등록하고 각각 이슈를 할당받아 진행하기로 했다. 
목표는
- 프로젝트 히스토리 파악
- 리뷰어 찾기
- 리뷰 받기

모임 시간 안에 패치 업로드까지 하는 것까지 진행을 했다. 모임 시간 안에 끝내지 않으면 집에 가서 언제 할지 모르고 나도 직접 봐주지 않으면 한명한명 대응해주기 어렵기 때문이었다. 오프라인/온라인 모임 시간 안에 모두가 올리고 마칠 수 있도록 도와줬다. 이후 업데이트는 각 이슈에 대한 내용을 적은 GitHub 이슈 댓글로 확인해줬다. 또 추가 질문이 있는 경우는 프로그램 디스코드 방이나 DM으로 진행을 했다. 

실제로 17명이 컨트리뷰션 실습 참여를 했고 리뷰 받는 중에 발생한 문제들을 정리해서 몇 번 공유를 했다. 나도 몰랐던 정책을 알게 된 부분도 있었다. 내가 소홀한 사이에 새로 추가된 Merge 조건도 있었다. Committer 의 권한이 알고 있던 것보다 좀 더 있다는 것도 알게 됐다. 그래서 다행히 멘티들이 패치를 반영하는데 도움을 줄 수 있었다. 

일단 개인적으로 기여하고 싶어서 찾아보고 있던 멘티들에게는 더 도움이 됐다. 가벼운 내용으로 한 번 진행해본 이후에 새 패치를 작업할 때 조금 더 쉽게 진행할 수 있었다. 그래서 잘한 일이라고 생각이 든다.
